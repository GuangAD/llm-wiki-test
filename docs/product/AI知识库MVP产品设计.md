# AI 知识库 MVP 产品设计

## 一句话定位

一个只服务个人写作与知识管理的 AI Native 本地知识库：你把链接、PDF、想法丢进来，它自动整理、编译、建立关联，并反向给你答案和选题。

## 目标

- 让零散素材有统一入口
- 让入库内容自动变成可用笔记
- 让相近内容自动连起来，不再靠人肉翻找
- 让知识库不仅能回答问题，还能反向给出写作选题

## 目标用户

单人作者自己，典型场景是公众号写作、个人知识沉淀、内容选题准备。

## 核心问题

- 素材散落：来源多，存放乱
- 检索低效：记得看过，但找不到
- 判断分散：看过很多内容，但观点没被提前编译
- 选题被动：知识库只能答，不会主动给建议

## MVP 边界

### 必须做

- 收一条链接、一份 PDF、或一段文本想法
- 自动生成标准化笔记
- 自动打标签并建立轻量关联
- 自动更新索引
- 支持问答、选题、周报三种输出

### 明确不做

- 多人协作
- 多端同步
- Web 前台
- 权限系统
- 音视频识别
- 配图和排版自动化
- 云端强依赖
- 向量库/RAG 作为首版前提

## 一页纸 MVP 定义

| 项目 | 结论 |
| --- | --- |
| 用户 | 我自己 |
| 问题 | 素材散落、写作时找不到、观点没被编译、选题效率低 |
| 输入 | 链接、PDF、Markdown、纯文本、灵感想法 |
| 处理 | 抓取原文 -> AI 摘要 -> 打标签 -> 建关联 -> 更新索引 -> 重编译主题页 |
| 输出 | 问答结果、可写选题、本周知识周报 |
| 不做什么 | 不做数据库、不做向量库、不做前端、不做协作 |

## 用户流程

### 流程 1：入库

1. 用户输入一个链接、文件路径或文本
2. 系统抓取正文或读取文件
3. AI 输出摘要、立场、标签
4. 系统落盘到 `raw/` 和 `notes/`
5. 系统计算关联，更新 `related_ids`
6. 系统更新索引和必要的主题页

### 流程 2：问答

1. 用户提问
2. 系统先查 `indexes/` 和 `wiki/`
3. AI 读取相关主题页和必要原子笔记
4. 输出带来源的回答

### 流程 3：选题

1. 用户执行选题命令
2. 系统读取 `wiki/` 中近期主题和空缺点
3. AI 输出 3-5 个可写方向，附理由和来源依据

### 流程 4：周报

1. 系统按周扫描新增条目
2. 汇总新增素材、热点主题、待编译主题
3. 生成本周知识周报到 `briefs/`

## 信息架构

### 目录结构

```text
knowledge/
  raw/
  notes/
  wiki/
  briefs/
  indexes/
  prompts/
  scripts/
  docs/
```

以上目录均相对于实际知识库工作区 `knowledge/`。

### 分层职责

- `raw/`：原始资料备份，只读保存，永不覆盖
- `notes/`：单条素材的 AI 笔记，是最小知识单元
- `wiki/`：多条笔记编译后的主题页，是问答主入口
- `briefs/`：选题、周报、专题清单
- `indexes/`：导航索引，帮助快速定位内容
- `prompts/`：固定提示词、工作流规范、标签体系
- `scripts/`：抓取、索引、关联、周报等机械脚本

## 模块设计

### 1. Ingest 模块

职责：把不同来源内容转成统一输入对象。

输入：URL、文件路径、纯文本。

输出：标准原文记录。

最小能力：

- 识别输入类型
- 提取正文
- 生成标准文件名和 `id`

### 2. Note 模块

职责：把单条输入转成可读笔记。

输出内容至少包含：

- 标题
- 三段式摘要
- 作者立场/我的判断
- 3-5 个标签
- 可能关联的话题

### 3. Relation 模块

职责：给新笔记找已有关联条目。

MVP 策略：

- 标签重叠分
- 标题关键词重叠分
- 分数超过阈值则建立双向关联

### 4. Compile 模块

职责：把原子笔记编译成主题页。首版只做主题页，不做实体页。

编译规则：

- 一个主题下至少有 2 条相关笔记才触发主题页
- 若新旧观点冲突，要显式写出冲突点
- 主题页必须附来源列表，不允许只留 AI 结论

### 5. Query 模块

职责：提供问答和定位。

MVP 策略：

- 先查索引，再查主题页
- 找不到主题页时回退到原子笔记
- 回答必须附来源文件或来源链接

### 6. Brief 模块

职责：输出选题和周报。

输出约定：

- 选题：标题、理由、引用素材、可写角度
- 周报：本周新增、重点主题、待整理清单

## 数据设计

### 统一 frontmatter 字段

```yaml
---
id: note_20260618_a8f31c92
content_id: cnt_20260618_a8f31c92
title: XXX
source_type: url
source_uri: https://example.com
created_at: 2026-06-18T10:00:00+08:00
captured_at: 2026-06-18T10:01:00+08:00
tags: [ai, writing]
summary: 一句话摘要
stance: approve
related_note_ids: []
topic_keys: []
status: active
---
```

`stance` 使用枚举值：

- `approve`
- `doubt`
- `neutral`
- `todo`

Phase 1 智能体生成结果的 `payload` 至少包含：

- `title`
- `summary`
- `tags`：3-5 个
- `stance`
- `key_points`：3-7 条
- `my_judgement`
- `useful_for`
- `related_topics`

### 文件命名规则

- ID 采用“日期 + 短 hash”，不使用自增序号
- `content_id`：`cnt_<YYYYMMDD>_<source_key_hash>`
- `note_id`：`note_<YYYYMMDD>_<source_key_hash>`
- `job_id`：`job_<YYYYMMDDHHMMSS>_<source_key_hash>`
- `request_id`：`gen_<YYYYMMDDHHMMSS>_<source_key_hash>_<generation_type>`
- `raw/`：`raw/YYYYMMDD/<content_id>.md` + `raw/YYYYMMDD/<content_id>.meta.yaml`
- `notes/`：`notes/YYYYMMDD/<note_id>.md`
- `wiki/`：`wiki/topic-<topic_key>.md`
- `briefs/`：`weekly-YYYY-WW.md` / `topic-picks-YYYYMMDD.md`

### 标签规则

- 标签数量控制在 3-5 个
- 标签来自固定标签表，不允许每次自由发挥
- 新标签进入标签表前，先人工确认是否重复

## 技术选型

### 首版推荐

- 语言：Python
- 存储：Markdown 文件 + frontmatter
- 运行：本地命令行
- AI：智能体参与生成，首版不接 LLM API
- 抓取：按来源用现成工具封装
- 检索：本地关键词搜索 + 索引文件

### 选型原则

- 主流：优先 Python 和成熟开源工具
- 本地：先本地跑通，不引入服务器
- 简单：文件优先，不上数据库
- 大众：优先社区大、AI 熟悉的工具

## 建议的首版命令

MVP 直接定义 4 个命令就够：

- `ingest <input>`：入库一条内容
- `ask <question>`：按知识库上下文问答
- `brief topics`：生成选题清单
- `brief weekly`：生成本周周报

这 4 个命令能完整覆盖输入、处理、输出闭环。

## 开发优先级

### Phase 1：最小闭环

- 实现 `ingest`
- 生成原始记录和单条笔记
- 更新 Markdown 索引：`recent.md / tags.md / sources.md`
- 实现扫描原子笔记的 `ask`

### Phase 2：知识可用

- 建立关联
- 生成主题页
- 实现主题页优先的 `ask`

### Phase 3：知识反向产出

- 实现 `brief topics`
- 实现 `brief weekly`

## 验收标准

- 1 分钟内完成一条素材入库
- 新增内容落盘后可在索引中被定位
- 任意一条结论都能追溯到原始来源
- Phase 1 问答至少能引用 2 条原子笔记
- Phase 2 问答优先引用主题页，不足时回退到原子笔记
- Phase 3 选题输出至少包含标题、理由、来源三项
- 连续使用 14 天后，仍愿意继续把新素材丢进来

## 风险与控制

- 抓取不稳定：按来源拆适配器，不做统一万能爬虫
- 标签失控：维护固定标签表，限制自由扩张
- 内容越积越乱：坚持分层，不允许把总结和原料混存
- AI 幻觉：所有结论必须保留来源链路

## V2 再考虑

- 多来源自动订阅
- Web 界面
- 移动端入口
- 更复杂的实体关系
- 向量检索或混合检索

## 直接开工结论

先实现 4 个命令、6 个目录、1 套 frontmatter 规范。只要第一周把 `ingest -> notes -> indexes -> ask` 跑通，这个项目就成立了。
