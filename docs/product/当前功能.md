# 当前功能

## 文档说明

- 文档日期：2026-07-16
- 适用版本：`fuxi-kb 0.1.0`
- 事实来源：`knowledge/kb/` 当前代码、`tests/` 自动化测试、实际 CLI 运行结果
- 功能口径：只记录已经实现的行为；设计文档中尚未落地的能力不计入当前功能

## 1. 目标

本项目是一个服务个人写作和知识管理的本地 AI 知识库。

它要解决的核心问题是：把 URL、PDF、Markdown、文本文件和临时想法统一归档，经过智能体生成结构化知识，再支持盘点、检索、主题编译、选题和周报输出。

系统由三部分组成：

1. `kb` CLI：执行可确定、可验证的文件操作和数据校验。
2. `knowledge/` 工作区：保存原文、笔记、主题页、索引、生成请求和任务状态。
3. Claude Code、Codex 等智能体：完成摘要、判断、主题页、选题和周报等语义生成，并根据 CLI 返回结果继续流程。

## 2. 当前范围

### 2.1 已实现

- 初始化知识库工作区
- 接收 URL、本地 PDF、Markdown、UTF-8 文本文件和直接输入的文本
- 保存原文及元数据
- 按正文哈希检测重复内容
- 创建智能体生成请求，接收并校验生成结果

- 自动计算轻量笔记关联

- 维护最近笔记、标签和来源索引
- 盘点知识库内容
- Wiki 优先、Note 回退的轻量关键词检索
- 通过两段式生成输出带来源的问答结果
- 用户显式选择时将问答沉淀为 Insight
- 执行确定性与语义知识 Lint 并保存报告

## 4. 核心流程

### 4.1 两段式生成

当前系统不在 CLI 内调用模型，而是把一次语义生成拆成两个阶段：

```text
用户或智能体执行 kb 命令
  -> CLI 保存确定性数据
  -> CLI 创建 GenerationRequest
  -> 返回 needs_generation
  -> 智能体读取来源和 prompt
  -> 智能体写入 GenerationResult YAML
  -> 执行对应的 --continue 命令
  -> CLI 校验结果并写入正式知识文件
```

这样可以把语义生成和文件写入分离：智能体负责“生成什么”，CLI 负责“写到哪里、格式是否正确、索引如何更新”。

### 4.2 入库流程

```text
输入识别
  -> 提取正文
  -> 计算正文 SHA-256
  -> 重复检测
  -> 写入 raw 正文和 meta.yaml
  -> 写入 job
  -> 创建 note GenerationRequest
  -> 智能体写入 GenerationResult
  -> kb ingest --continue
  -> 校验 note payload
  -> 写入 notes
  -> 建立关联和主题候选
  -> 必要时创建 topic GenerationRequest
  -> 重建 recent/tags/sources 索引
```

### 4.3 查询流程

```text
用户问题
  -> 优先检索 Topic 和 Insight
  -> 命中 Wiki 后按来源下钻 Note
  -> 无 Wiki 命中时回退 Note
  -> 创建 answer GenerationRequest
  -> 智能体生成带引用的 Answer Result
  -> kb ask --continue 校验并返回答案
  -> 首次命令带 --save 时写入 Insight
```

### 4.4 主题页流程

```text
新笔记落盘
  -> 标准化 related_topics 为 topic_keys
  -> 统计同一 topic_key 下的笔记数量
  -> 数量达到 2 条时创建 topic GenerationRequest
  -> 智能体写入 topic GenerationResult
  -> kb compile --continue
  -> 写入 wiki/topic-<topic_key>.md
  -> 重建 indexes/topics.md
```

### 4.5 Brief 流程

选题和周报均采用两段式生成：

```text
kb brief topics 或 kb brief weekly
  -> 收集索引、主题页和笔记路径
  -> 创建 GenerationRequest
  -> 智能体写入 GenerationResult
  -> 对应 --continue 命令
  -> 写入 briefs/
```

## 5. 命令功能

### 5.1 `kb init`

执行方式：

```powershell
uv run kb init
```

功能：

- 创建 `raw/`、`notes/`、`wiki/`、`briefs/`、`indexes/`、`prompts/`、`state/` 和 `logs/` 等运行目录
- 创建 `note.md`、`topic.md`、`brief-topics.md`、`brief-weekly.md` 默认 prompt
- 已存在的 prompt 不会被覆盖
- 已有知识内容不会被清空

成功状态为 `completed`。

### 5.2 `kb ingest <input>`

执行方式：

```powershell
uv run kb ingest "<input>"
```

输入识别规则：

| 条件 | 输入类型 | 处理方式 |
| --- | --- | --- |
| 以 `http://` 或 `https://` 开头 | URL | `httpx` 下载，`trafilatura` 提取正文；提取失败时使用原始响应文本 |
| 输入值对应已存在文件 | 本地文件 | PDF 使用 PyMuPDF；其他文件按 UTF-8 文本读取 |
| 其他情况 | 直接文本 | 原样作为正文 |

首次执行会：

- 对正文计算 SHA-256
- 在已有 `raw/**/*.meta.yaml` 中查找相同正文哈希
- 重复时返回 `duplicate`，不创建新笔记
- 非重复时保存原文和元数据
- 创建 job、note 生成请求和预期结果路径
- 返回 `needs_generation`

主要落盘文件：

```text
raw/YYYYMMDD/<content_id>.md
raw/YYYYMMDD/<content_id>.meta.yaml
state/jobs/<job_id>.json
state/generation_requests/<job_id>-note.md
state/generation_results/<job_id>-note.yaml
```

### 5.3 `kb ingest --continue <job_id>`

执行方式：

```powershell
uv run kb ingest --continue <job_id>
```

兼容别名：

```powershell
uv run kb ingest --continue-job <job_id>
```

功能：

- 检查 job 是否存在
- 读取并校验 note 生成结果
- 写入原子笔记
- 更新双向关联关系
- 计算主题候选
- 必要时创建 topic 生成请求
- 重建 `recent.md`、`tags.md`、`sources.md`
- 更新 job 为 `completed`

### 5.4 `kb status`

执行方式：

```powershell
uv run kb status
```

返回内容：

- `raw`、`notes`、`wiki`、`topics`、`insights`、`briefs`、`reports` 的 Markdown 文件数量
- `indexes/recent.md` 中的笔记标题和路径
- `indexes/tags.md` 中的标签和笔记 ID
- `indexes/sources.md` 中的来源 URI 和笔记 ID
- 实际读取的索引路径

该命令适合回答“当前有哪些内容、多少笔记、有哪些标签、最近入库了什么”等盘点问题。

### 5.5 `kb ask <question>`

执行方式：

```powershell
uv run kb ask "<question>"
uv run kb ask "<question>" --save
uv run kb ask --continue <job_id>
```

首次执行优先检索 Wiki，并创建 `answer` GenerationRequest；智能体只能引用请求提供的来源。继续执行会校验问题、请求身份和引用路径。只有首次命令带 `--save` 时才写入 `wiki/insight-*.md` 和 `indexes/insights.md`。

### 5.6 `kb compile --continue <job_id> --topic-key <topic_key>`

执行方式：

```powershell
uv run kb compile --topic-key <topic_key>
uv run kb compile --continue <job_id> --topic-key <topic_key>
```

功能：

- 首次执行扫描至少 2 条同主题 Note，并把已有 Topic 作为旧版本参考
- 读取 topic 生成结果
- 校验 payload 类型
- 校验结果中的 `topic_key` 与命令参数一致
- 写入主题页
- 重建主题索引

输出文件：

```text
wiki/topic-<topic_key>.md
indexes/topics.md
```

Topic 请求既可由 `ingest --continue` 自动触发，也可通过首次编译命令主动创建。

### 5.7 `kb brief topics`

首次执行：

```powershell
uv run kb brief topics
```

继续执行：

```powershell
uv run kb brief topics --continue <job_id>
```

首次执行会把 `recent.md`、`tags.md`、存在时的 `topics.md`、主题页和原子笔记路径写入生成请求。继续执行会校验生成日期与 job 目标日期一致，并生成：

```text
briefs/topic-picks-YYYYMMDD.md
```

每个选题包含：

- 标题
- 选题理由
- 写作角度
- 来源笔记 ID

### 5.8 `kb brief weekly`

首次执行：

```powershell
uv run kb brief weekly
```

继续执行：

```powershell
uv run kb brief weekly --continue <job_id>
```

系统按当前 ISO 周生成目标周标识，例如 `2026-W29`。继续执行会校验结果周次，并生成：

```text
briefs/weekly-YYYY-Www.md
```

周报包含：

- 本周新增
- 重点主题
- 未决问题
- 下一步行动

### 5.9 `kb lint`

```powershell
uv run kb lint
uv run kb lint --continue <job_id>
```

首次执行检查损坏链接、重复身份、缺失来源和未进入索引的 Wiki，同时创建语义检查请求。继续执行写入 `reports/lint-*.md`，不自动修改知识文件。

## 6. 数据设计

### 6.1 ID 和文件命名

| 对象 | 格式 |
| --- | --- |
| 原文 ID | `cnt_<YYYYMMDD>_<8位hash>` |
| 笔记 ID | `note_<YYYYMMDD>_<8位hash>` |
| job ID | `job_<YYYYMMDDHHMMSS>_<8位hash>` |
| note/brief request ID | `gen_<YYYYMMDDHHMMSS>_<8位hash>_<generation_type>` |
| topic request ID | `gen_<YYYYMMDDHHMMSS>_<topic_key>_topic` |

原文 ID 和笔记 ID 使用同一来源键哈希。重复检测不依赖来源 URI，而是依赖正文 SHA-256。

### 6.2 原文元数据

`raw/**/*.meta.yaml` 当前保存：

| 字段 | 含义 |
| --- | --- |
| `id` | `content_id` |
| `source_type` | `url`、`pdf`、`md`、`txt` 或 `text` 等 |
| `source_uri` | URL、本地绝对路径或 `text:` 前缀文本标识 |
| `source_title` | 文件名、URL 或输入文本截断标题 |
| `captured_at` | 抓取时间 |
| `content_hash` | 正文 SHA-256 |
| `mime_type` | 内容 MIME 类型 |
| `status` | 当前固定为 `active` |

### 6.3 Note 生成结果

完整 GenerationResult YAML 除 `payload` 外还包含：

| 字段 | 规则 |
| --- | --- |
| `request_id` | 生成请求 ID |
| `job_id` | 所属 job ID |
| `content_id` | note 流程使用的原文 ID；模型层当前未强制 note 必填 |
| `generation_type` | `note`、`topic`、`brief_topics` 或 `brief_weekly` |
| `status` | 当前只接受 `completed` |
| `created_at` | 生成时间字符串 |
| `sources` | 来源路径及可选 `source_uri` 列表 |
| `payload` | 与 `generation_type` 匹配的结构化内容 |

`note_v1` payload 的校验规则：

| 字段 | 规则 |
| --- | --- |
| `title` | 非空字符串 |
| `summary` | 非空字符串 |
| `tags` | 3 至 5 个字符串 |
| `stance` | `approve`、`doubt`、`neutral`、`todo` |
| `key_points` | 3 至 7 条 |
| `my_judgement` | 非空字符串 |
| `useful_for` | 字符串列表 |
| `related_topics` | 字符串列表 |

正式笔记使用 Markdown frontmatter，正文固定包含摘要、关键信息、我的判断、可用场景和主题候选。

### 6.4 关联规则

新笔记与已有笔记逐一计算关联分：

```text
关联分 = 共同标签数量 × 4 + 共同标题词数量 × 2
```

分数达到 6 时：

- 双方的 `related_note_ids` 都写入对方笔记 ID
- 不重复添加已有 ID

智能体输出的 `related_topics` 会被转换为小写、连字符形式的 `topic_keys`。同一 `topic_key` 至少出现在 2 条笔记中时，系统认为该主题可编译。

### 6.5 Topic 生成结果

`topic_v1` payload 包含：

- `topic_key`
- `title`
- `definition`
- `conclusions`
- `disagreements`
- `extensions`
- `source_note_ids`，至少 2 条

主题页正文固定包含主题定义、主要结论、分歧或未决问题、延伸方向和来源笔记列表。

### 6.6 Brief 生成结果

选题 payload：

- `date`
- 一个或多个 `topics`
- 每个 topic 包含 `title`、`reason`、`angle`、`source_note_ids`

周报 payload：

- `week`
- `new_items`
- `key_themes`
- `open_questions`
- `next_actions`

### 6.7 CLI 响应协议

CLI 正常输出为单行 JSON。常见字段包括：

- `ok`
- `command`
- `status`
- `job_id`
- `next_action`
- 生成请求、生成结果或正式文件路径
- `message`

业务错误同样以 JSON 返回，包含 `error_code`、`error_message` 和 `retryable`。当前业务错误通常仍以进程退出码 `0` 返回，自动化调用方必须以 `ok` 和 `status` 判断结果，不能只看退出码。

## 8. 当前实现边界与已知限制

### 8.1 配置文件尚未接入主要运行流程

`knowledge/kb.yaml` 已定义路径、允许输入、入库锁和查询参数，配置加载器也已实现；但当前 CLI 服务没有调用 `load_config()`。

因此：

- 主要目录仍由代码硬编码
- `ask.max_notes` 和 `ask.min_score` 当前不会从配置文件生效
- `ingest.allowed_inputs` 当前不会限制实际输入
- `ingest.lock_enabled` 当前不能关闭入库锁

### 8.2 查询仍是轻量关键词召回

当前 `ask` 已支持 Wiki 优先和中文二元词组匹配，但仍不支持 Embedding、同义词扩展和语义重排。知识规模明显增长后再评估混合检索。

### 8.3 “最近笔记”是索引名称，不是严格时间窗口

`indexes/recent.md` 当前包含全部笔记，按文件路径排序写入，没有数量上限，也没有显式倒序。因此 `status.recent_notes` 返回的是该索引中的全部条目，不保证最新优先。

### 8.4 Lint 不自动修复

`kb lint --continue` 只生成报告。主题修复必须显式执行 `kb compile --topic-key <topic_key>`，避免语义检查直接重写知识文件。

### 8.5 Topic 请求兼容旧路径

新 Topic 请求使用 `<job_id>-topic-<topic_key>` 唯一路径；读取层仍兼容旧的 `<job_id>-topic` 路径。

### 8.6 生成结果一致性校验

当前校验：

- GenerationResult schema
- `generation_type` 与 payload 类型
- topic 的 `topic_key`
- brief 的目标日期或目标周次
- brief job 的生成类型
- request/result 的 `request_id`、`job_id`、`generation_type`、`content_id` 和来源范围
- Answer 引用必须属于请求提供的来源

### 8.7 来源类型在正式笔记中会被简化

原文元数据会保留真实 `source_type`，但正式 note frontmatter 当前把 `source_type` 固定写为 `text`，`source_uri` 取生成结果中的第一条 source，而不是直接从 raw 元数据复制。

### 8.8 异常输出尚未完全统一

job 不存在、生成结果缺失或 schema 错误会返回结构化错误 JSON；但 URL 请求失败、文件编码错误、PDF 解析错误等适配器异常尚未统一转换为标准错误响应。

## 9. 验收标准

当前版本按以下标准判断功能可用：

| 功能 | 验收结果 |
| --- | --- |
| 初始化 | 能创建所有运行目录和默认 prompt，且不覆盖已有 prompt |
| 文本入库首次执行 | 能保存 raw、job 和 generation request，并返回 `needs_generation` |
| 入库继续执行 | 能校验结果、写入 note、更新索引和 job |
| 重复检测 | 相同正文不重复入库 |
| 关联 | 达到阈值时能写入双向 `related_note_ids` |
| 主题 | topic 候选达到 2 条笔记时能生成请求，继续执行后能写主题页和索引 |
| 盘点 | 能返回内容数量、最近笔记、标签和来源 |
| 查询 | 能优先检索 Wiki、生成带来源回答，并按显式选项保存 Insight |
| Lint | 能输出确定性与语义问题报告，且不自动修改 Wiki |
| 选题 | 能完成 request/result/continue 流程并写入选题清单 |
| 周报 | 能校验目标周次并写入周报 |
| 工程质量 | 自动化测试通过，Ruff 检查通过 |
